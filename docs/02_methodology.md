# 02. 推荐开发方式与理由

## 推荐方式：增量开发（Incremental Development）

**结论**：对于这个单人独立开发项目，**增量开发**是最优选择。

## 什么是增量开发

增量开发的核心理念是：把整个产品拆分成多个小的、独立的功能模块（增量），一个版本实现一个增量，每个版本结束时都有一个**完整可运行的产品**。

每个版本都在前一个版本的基础上增加新功能，而不是等到所有功能做完才一起发布。

类比：就像搭乐高，先搭底座（V0），然后搭第一层（V1），然后第二层（V2）……每一层搭完都是一个完整的、可以玩的成品，而不是等到全部搭完才能看。

## 为什么增量开发最适合这个项目

### 1. 完美匹配"每个版本都能看到明确成果"的需求
- 你希望每个版本都从最小可实现效果开始，每一步都有明确成果
- 增量开发每个版本结束都有完整可运行的产品，随时可以切 tag 玩
- V0 就能跑起来，V1 能看棋盘，V2 能走子，V3 有规则……每一步都有实感

### 2. 单人开发，风险可控
- 独立开发者时间精力有限，不需要复杂的流程和仪式
- 每个版本只做一件事，聚焦，不容易乱
- 即使中途有事暂停，也有完整的可用版本，不会烂尾
- 出问题了可以快速回滚到上一个稳定版本

### 3. 通用层和业务层可以解耦推进
- 先做通用脚手架层（V0.x），再做象棋业务层（V1.x+）
- 通用层做完可以验证复用性，后续做其他游戏直接用
- 不会一开始就把架构绑死在象棋上

### 4. 反馈周期短，随时调整方向
- 每个版本做完都可以玩一玩，感受一下，有问题及时调整
- 不会走太远才发现方向错了
- 做着做着想加功能或者改设计，都可以放到下一个增量里

### 5. 符合独立开发的节奏
- 不需要站会、sprint 规划、回顾会议这些团队流程
- 想做多久做多久，想停就停
- 进度一目了然，每个版本打 tag，清晰可追溯

## 为什么不选其他方式

| 方式 | 不适合的原因 |
|------|-------------|
| 瀑布 | 太僵化，需求不能变，独立开发很容易中途改想法，瀑布模式改起来成本太高 |
| 螺旋 | 太重了，适合大型高风险项目，有复杂的风险分析流程，单人小游戏没必要 |
| 迭代开发 | 也不错，但迭代更偏向"反复改进同一批功能"，增量更偏向"逐步增加新功能"，我们的场景更适合增量 |
| 原型法 | 适合需求非常不明确的项目，这个项目需求相对明确（做象棋，然后做通用框架），不需要专门做原型验证 |
| RAD（快速应用开发） | 依赖大量代码生成工具和可复用组件，我们是自己造轮子，不适用 |
| JAD（联合应用开发） | 需要多方用户参与讨论，单人项目不存在这个场景 |
| 敏捷 | 概念太泛，是一个大的方法论集合，不是具体的开发方式，而且通常是团队用的 |
| XP（极限编程） | 有结对编程、测试驱动、持续集成等实践，单人开发大部分实践用不上，太重 |
| Scrum | 有 Product Owner、Scrum Master、站会、sprint、回顾会……都是团队协作的流程，单人开发纯属于给自己加戏 |
| 精益开发 | 理念很好，核心是消除浪费，但更侧重生产过程优化，不如增量开发直接贴合"逐个版本加功能"的实际操作 |
| DevOps | 侧重开发运维一体化、CI/CD 这些，我们这个项目暂时还到不了这个阶段，后面可以加，但不是核心开发方式 |

## 增量开发在本项目中的具体落地

### 版本划分原则
1. 每个版本只做**一个核心主题**，不贪多
2. 每个版本结束时都是**完整可运行**的产品
3. 通用层和业务层分开推进，先搭架子再填内容
4. 从小到大，从核心到外围

### 版本节奏
- V0.x：基础脚手架（通用层，和游戏无关）
- V1.x：棋盘与棋子（可视化）
- V2.x：规则引擎（逻辑层）
- V3.x：UI 与系统（交互层）
- V4.x：音效与动效（体验层）
- V5.x：AI 与发布（完整成品）

### 每个版本的固定流程
1. 明确本版本的目标和范围
2. 实现代码
3. 测试可运行
4. 写文档和 changelog
5. 打 tag
6. 进入下一个版本

## 补充：结合一点精益思想

虽然主选增量开发，但可以吸收精益开发的一些好的实践：
- **消除浪费**：不做暂时用不上的功能，不提前过度设计
- **小步快跑**：每个版本尽量小，尽快交付
- **持续改进**：每个版本做完复盘一下，哪里可以更好，下个版本优化

不搞复杂流程，只取好用的部分。
