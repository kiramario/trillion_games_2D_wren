# 文档规范

## 一、文档类型与存放位置

| 文档类型 | 存放位置 | 说明 |
|----------|----------|------|
| 项目核心文档 | docs/*.md | 愿景、架构、路线图等 |
| 开发日志 | docs/devlog/ | 每个版本一个开发日志 |
| 接口文档 | 跟代码放一起，或docs/api/ | 模块API说明 |
| 设计文档 | docs/design/ | 功能设计方案 |
| 美术资源说明 | assets/ 对应目录 | 资源说明 |
| 测试文档 | tests/ | 测试用例、测试报告 |
| README | 项目根目录 | 项目概览、快速开始 |

---

## 二、文档写作规范

### 2.1 语言
- 正文用**中文**写
- 专业术语可以用英文，第一次出现时可以加中文解释
- 代码、变量名、文件名用英文，保持跟代码一致

### 2.2 格式
- 统一用 Markdown 格式
- 文件名：全小写+下划线，如 `architecture.md`
- 标题层级：# 一级，## 二级，### 三级，最多四级
- 段落之间空一行

### 2.3 结构
每个文档开头要有：
- 文档标题
- 一句话简介
- （可选）版本号、最后更新日期

每个文档结尾要有：
- 文档版本号
- 更新日期（可选）

---

## 三、各类型文档要求

### 3.1 README.md
**位置：项目根目录
**内容要求：
1. 项目名称 + 一句话介绍
2. 功能特性（ bullet list
3. 快速开始：怎么运行、怎么构建
4. 项目结构说明
5. 开发进度 / 当前版本
6. 联系方式 / 作者信息

### 3.2 设计文档
**位置：docs/design/xxx_design.md
**内容要求：
1. 背景：为什么要做这个功能
2. 目标：这个功能要达到什么效果
3. 方案设计：具体怎么实现
   - 架构图/流程图（能用文字就用文字，复杂的再画图）
   - 模块划分
   - 接口定义
4. 备选方案：考虑过的其他方案，为什么不选
5. 风险评估：可能的问题和应对
6. 验收标准：怎么算做完了

### 3.3 开发日志
**位置：docs/devlog/v0.0.1.md
**命名：版本号.md
**内容要求：
1. 版本号 + 发布日期
2. 本版本做了什么（feature list）
3. 技术要点 / 实现思路
4. 踩过的坑 / 遇到的问题和解决方案
5. 遗留问题 / TODO
6. 下一步计划

**为什么要写开发日志？
- 过几个月回来看，知道当时是怎么想的
- 踩过的坑不要再踩第二次
- 复盘用，看看自己的开发节奏

### 3.4 API 文档
**位置：跟代码放一起，或docs/api/
**内容要求：
每个模块/类的文档包含：
1. 模块功能说明
2. 主要方法列表
   - 方法名
   - 参数说明
   - 返回值说明
   - 功能描述
3. 使用示例
4. 注意事项

## 四、代码注释规范

### 4.1 文件头注释
每个Lua文件开头都要有：
```lua
--[[
    文件名：xxx.lua
    功能：一句话说明这个文件是干嘛的
    作者：wren
    创建日期：2026-06-09
    依赖：core.logger, core.config
]]
```

### 4.2 模块/类注释
每个模块/类定义的前面要有说明：
```lua
--[[
    日志系统模块
    功能：提供分级日志输出，支持调试/信息/警告/错误四个级别
    用法：
        local Logger = require("core.logger")
        Logger.info("游戏启动")
        Logger.error("出错了")
]]
local Logger = {}
```

### 4.3 函数注释
每个公有函数前面要有注释：
```lua
--[[
    记录信息级别的日志
    参数：
        msg (string) - 日志消息
    返回：无
    示例：Logger.info("玩家进入游戏")
]]
function Logger.info(msg)
    -- ...
end
```

### 4.4 行内注释
- 复杂逻辑旁边加注释，说明为什么这么写
- 魔法数字旁边加注释，说明这个数字是啥意思
- Lua特有语法旁边加注释，类比JS/Java/Python帮助理解
- 不要写废话，不要解释"代码在做什么"，要解释"为什么这么做"

```lua
-- ✅ 好的注释
-- 除以2是因为要从中心点绘制，锚点在中心
local x = screen_width / 2

-- ❌ 坏的注释
-- x 等于屏幕宽度除以2
local x = screen_width / 2
```

### 4.5 Lua 特性注释（对初学者友好）
凡是涉及Lua特有的语法、API、或者跟你熟悉的语言不一样的地方，都要加类比注释。

常见需要加注释的地方：
- metatable / 继承
- 冒号语法 / self
- 闭包 / upvalue
- table的各种用法
- LÖVE2D的特殊API
- 协程（如果用到的话）

示例：
```lua
-- __index 元方法：当访问table不存在的key时，会调用这个
-- 类比 JS 的原型链查找，或 Java 的父类方法继承
setmetatable(child, {__index = parent})

-- 冒号语法：自动把self作为第一个参数传进去
-- 类比 Python 的 self 参数，JS 的 this
function Player:move(dx, dy)
    self.x = self.x + dx
end
-- 等价于 function Player.move(self, dx, dy)
```

## 五、文档维护规范

### 5.1 代码和文档同步更新
- 改了代码，对应的文档也要改
- 不要让文档过期，过期的文档比没有还不如没有
- 提交代码的时候，一起提交文档更新

### 5.2 文档版本
- 重要文档末尾加版本号
- 大版本更新时，更新文档版本号
- 可以在版本号旁边可以参考语义化版本

### 5.3 图片
- 尽量用文字描述，能不用图就不用图（文字好维护
- 必须用图的话，放在docs/images/目录下
- 图片格式用png或svg

## 六、注释多少合适？

**原则：**
- 给3个月后的自己看，能不能看懂
- 给一个刚接触Lua的人看，能不能看懂
- 不要太少，也不要太多，恰到好处
- 复杂的地方多写，简单的地方少写
- 为什么 业务逻辑多写，通用代码少写

**判断标准：**
如果你写这段代码的时候想了超过5分钟以上，就加个注释，把思路写下来。下次再看的时候不用再想5分钟。
