# Trillion Games 2D - 通用2D游戏引擎脚手架 & 中国象棋

## 项目愿景

### 一句话定位
一个独立开发者友好的、可复用的2D游戏开发脚手架，第一个落地产品是中国象棋，后续可快速扩展到弹珠、卡牌、棋牌等各类2D游戏。

### 核心目标
1. **技术可控**：架构清晰、模块边界明确、注释详尽，独立开发者完全掌控每一行代码
2. **快速复用**：通用层与业务层严格分离，新游戏只需编写业务逻辑，基础能力开箱即用
3. **视觉品质**：2D画面但有纵深感，通过分层、光影、粒子、动画营造高级感，不碰真3D
4. **多端发布**：一次开发，优先Steam，后续支持Android APK、Windows/macOS/Linux桌面版
5. **迭代友好**：每个版本都有明确可验证的产出，随时可以停下来，随时可以继续

### 第一个产品：中国象棋
- 经典中国象棋规则
- 人机对战 + 本地双人对战
- 多种难度AI
- 存档/读档
- 悔棋功能
- 精美棋盘棋子视觉效果
- 走法提示、将军提示、胜负判定

### 后续产品方向
- 弹珠台游戏
- 卡牌对战
- 休闲益智类
- 更多棋类游戏

## 推荐开发方式：**增量式开发 (Incremental Development)**

### 为什么选增量式，而不是其他方式？

#### ❌ 不选瀑布模型
- 原因：需求会变，独立开发者容易在前期规划上浪费太多时间，很久看不到成果，容易弃坑。

#### ❌ 不选螺旋模型
- 原因：太重了，风险分析环节对于单人小项目是过度设计。

#### ❌ 不选原型法 (Throwaway Prototype)
- 原因：原型写完就扔太浪费，我们每一行代码都要沉淀下来变成可复用的资产。

#### ❌ 不选敏捷/Scrum/XP
- 原因：这些都是团队协作方法论，单人项目搞站会、故事点、Sprint回顾是自欺欺人，徒增 overhead。

#### ❌ 不选精益/DevOps
- 原因：更偏向生产环境和大规模团队，独立游戏开发用不上。

#### ❌ 不选RAD/JAD
- 原因：需要快速工具生成和用户联合设计，不适合单人独立开发。

#### ✅ 为什么选增量式开发？
1. **每个版本都有可运行的成果**：V0就能跑起来，每加一个版本就能看到新功能，正反馈强，不容易弃坑
2. **风险前置**：基础架构先做，技术风险早暴露，越往后越稳
3. **可随时暂停/恢复**：每个版本都是完整可运行的，忙的时候放三个月，回来接着做不用重新理
4. **架构自然演化**：先做最核心的，需要的时候再加，不会过度设计
5. **适合单人节奏**：不用跟任何人对齐，自己掌握速度，想快就快想慢就慢

### 增量式开发的具体执行方式
- 每个版本（V0, V1, V2...）都是**完整可运行**的
- 每个版本只加**一个核心能力**
- 每个版本结束都打tag，随时可以回退
- 每个版本都有明确的验收标准
- 通用层优先，业务层后加；基础能力优先，特效表现后加

## 整体架构

### 分层架构（从下到上）

```
┌─────────────────────────────────────────┐
│         业务层 (Game Specific)          │
│  中国象棋规则 / 弹珠逻辑 / 卡牌系统     │
│  - 棋盘、棋子、走法、AI                 │
├─────────────────────────────────────────┤
│         场景层 (Scene Layer)            │
│  菜单 / 游戏 / 设置 / 结算 等场景切换   │
│  - SceneManager, BaseScene              │
├─────────────────────────────────────────┤
│         系统层 (Systems Layer)          │
│  渲染 / 输入 / 音效 / UI / 粒子         │
│  - RenderSystem, InputSystem            │
│  - AudioSystem, UISystem, ParticleSystem│
├─────────────────────────────────────────┤
│         核心层 (Core Layer)             │
│  资源管理 / 配置 / 日志 / 事件总线      │
│  - ResourceManager, Config, Logger      │
│  - EventBus, StateMachine               │
├─────────────────────────────────────────┤
│         LÖVE2D 引擎层                   │
│  love.graphics / love.audio / etc.      │
└─────────────────────────────────────────┘
```

### 核心设计原则
1. **依赖方向永远向下**：上层可以调用下层，下层绝对不能知道上层的存在
2. **通用与业务严格分离**：core/systems/scenes层都是通用的，换游戏不用改
3. **面向接口编程**：每个系统都有明确的API，内部实现可以随时替换
4. **事件驱动解耦**：模块之间通过EventBus通信，不直接互相调用
5. **配置驱动**：可调参数都放在配置文件里，不用改代码就能调参

**目标**：把通用底层全部跑通，能打开窗口、切换场景、有日志有配置，跟具体游戏无关。

**范围**：

- 事件总线
- 场景管理器 + 2个空场景（菜单、游戏）
- 输入系统（键盘+鼠标事件封装）
- 渲染分层（背景层/游戏层/UI层/调试层）

**代码文件**：
- conf.lua
- src/main.lua
- src/core/logger.lua
- src/core/config.lua
- src/core/event_bus.lua
- src/core/resource_manager.lua
- src/core/utils.lua

- src/scenes/scene_manager.lua
- src/scenes/base_scene.lua
- src/scenes/menu_scene.lua
- src/scenes/game_scene.lua

**验收标准**：
1. 执行 love . 能正常打开窗口，不报错
2. 窗口标题、大小、背景色跟配置一致
3. 按1/2键可以在菜单场景和游戏场景之间切换
4. 控制台有清晰的日志输出（INFO/WARN/ERROR分级）
5. 鼠标点击/键盘按键有日志输出
6. 场景切换时有事件触发日志
7. 渲染四层都能看到不同颜色的测试内容

**为什么先做这些**：
这是所有游戏的地基，越早搭好，后面写业务越爽。而且这些都是通用能力，下个游戏直接用。

### V0.1.0 - 棋盘渲染版
**目标**：画出中国象棋棋盘，有基本的视觉效果，开始有游戏感。

**范围**：
- 棋盘数据结构
- 棋盘渲染（木纹背景、网格线、楚河汉界）
- 坐标系统（棋盘坐标 <-> 屏幕坐标转换）
- 简单光影效果（边缘暗角、棋盘纹理）
- 相机系统（缩放、平移，先做接口，后续用）

**代码文件**：
- src/entities/board.lua
- src/systems/camera_system.lua
- assets/images/ 相关资源（先用程序生成，后续换美术）

**验收标准**：
1. 进入游戏场景能看到完整的中国象棋棋盘
2. 棋盘居中显示，比例正确
3. 鼠标移动到棋盘上，能在日志里输出对应的棋盘坐标

**为什么先做棋盘**：
棋盘是静态的，先把画面基底搭好，后面放棋子才有地方放。而且坐标系统是整个游戏的基础，越早定越好。

**目标**：加入AI，可以跟电脑下棋。

**为什么这一版加AI**：
有了AI，一个人也能玩，游戏的可玩性大大提升。而且AI是独立模块，不影响现有规则，风险低。

## 代码规范

### Lua 代码风格
1. **缩进**：4个空格，不用tab
2. **命名**：
   - 变量/函数：snake_case（小写下划线）
   - 类/模块：PascalCase（大驼峰）
   - 常量：UPPER_SNAKE_CASE（全大写下划线）
   - 私有变量：_下划线开头
3. **文件**：全小写+下划线，跟里面的主类/模块对应
4. **注释**：
   - 每个文件开头有文件说明
   - 每个函数有功能说明、参数说明、返回值说明
   - 涉及Lua特性或LÖVE2D API的地方，加JS/Java/Python类比注释
   - 复杂逻辑加行内注释
5. **代码组织**：
   - 每个文件一个主要类/模块
   - 先写import，再写常量，再写类定义，最后return
   - 函数按功能分组，相关函数放一起

### 架构规范
1. **分层严格**：core层不能引用systems，systems不能引用scenes，scenes不能引用entities里的具体业务
2. **依赖注入**：需要什么依赖从参数传进来，不要在模块内部require上层模块
3. **事件驱动**：模块之间通过EventBus通信，不要直接调用
4. **配置外置**：所有可调参数都放config里，不要硬编码
5. **错误处理**：重要操作都有pcall保护，出错打日志不要崩

## 发布路径建议

### 桌面版 (Windows/macOS/Linux) - 最先做
**工具**：love-release 或 手动打包
**流程**：
1. 把代码打成 .love 文件（就是zip改后缀）
2. 跟LÖVE2D的可执行文件合并
3. 打包成安装包（Windows用Inno Setup，macOS用dmg）
**难度**：⭐ 最简单，LÖVE2D原生支持

### Steam 版 - 优先级最高
**基于桌面版**：
1. 做好桌面版
2. 申请Steam开发者账号（100美元）
3. 接入Steamworks Lua库（luasteam）
4. 上传到Steam后台
5. 审核发布
**难度**：⭐⭐ 主要是流程麻烦，技术上不难

### Android APK 版 - 后续做
**工具**：LOVE for Android 或 自己编译
**流程**：
1. 用官方提供的Android构建模板
2. 把.love文件放进去
3. 用Android Studio编译成APK
4. 适配触屏操作
**难度**：⭐⭐⭐ 需要改一下输入适配，其他还好

### 注意事项
1. **先做桌面版**：开发调试都方便，验证玩法最快
2. **Steam优先**：变现能力最强，独立游戏主要收入来源
3. **Android后做**：触屏适配需要单独做，而且变现难，先放一放
4. **iOS不建议做**：签名麻烦，审核严，独立开发者没必要折腾

## 主要风险和替代方案

### 风险1：LÖVE2D性能不够做复杂游戏
**概率**：低  **影响**：中
**应对**：
- 目前做象棋和弹珠完全够用
- 真不够了可以用LuaJIT，性能提升很多
- 再不够可以把核心逻辑用C写，Lua调用
- 极端情况换Godot，但目前完全没必要

### 风险2：伪3D视觉效果实现成本太高
**概率**：中  **影响**：低
**应对**：
- 先做纯2D版本，保证玩法
- 视觉效果逐步加，一版加一点
- 优先做成本低效果好的：阴影、分层、粒子
- 透视、镜头这些成本高的，有时间再加，没有也不影响玩

### 风险3：AI难度做不好，要么太简单要么太难
**概率**：中  **影响**：中
**应对**：
- 先做多个难度档，覆盖不同水平
- 加入随机因素，让AI不要每次都走一样的
- 后续可以接入开源象棋引擎，直接用现成的
- 实在不行就主打双人对战

### 风险4：Steam发布审核不通过
**概率**：低  **影响**：中
**应对**：
