# 05. 版本路线图（V0-V5 详细迭代计划）

## 版本规划总览

采用增量开发，每个版本只做一个核心主题，每个版本结束都有完整可运行的成果。

| 版本 | 核心主题 | 定位 | 预计工作量 |
|------|----------|------|-----------|
| V0.0.0 | 项目初始化 | 文档和规划 | ★☆☆☆☆ |
| V0.1.0 | 基础脚手架 | 通用核心模块 | ★★☆☆☆ |
| V0.2.0 | 渲染增强 | 渲染/相机/实体/动画 | ★★★☆☆ |
| V1.0.0 | 棋盘与棋子 | 可视化第一版 | ★★★☆☆ |
| V2.0.0 | 规则引擎 | 核心玩法逻辑 | ★★★★☆ |
| V3.0.0 | UI 与系统 | 交互与存档 | ★★★☆☆ |
| V4.0.0 | 音效与动效 | 体验打磨 | ★★★☆☆ |
| V5.0.0 | AI 与发布 | 完整可发布版本 | ★★★★☆ |

2. 所有规划文档都写完，内容完整
3. Git 仓库初始化完成，有第一个 commit
4. 打了 v0.0.0 的 tag
5. 目录结构和规划的一致

### 💡 为什么这一版先做这些

- 独立开发很容易写着写着就乱了，先有文档约束方向
- 第一个 tag 作为起点，后面所有版本都可以回溯到这里

## V0.1.0：基础脚手架

### 🎯 目标
- 把通用框架的核心基础模块跑通
- 完成"启动 → 加载 → 场景切换 → 退出"的完整链路
- 这一版和具体游戏完全无关，是纯通用框架

### 📦 范围
- 日志系统
- 配置系统
- 输入管理
- 场景管理
- 资源管理
- 工具函数
- 两个测试场景（验证场景切换）

### 📁 涉及模块/文件
```
src/core/
├── logger.lua           # 日志系统
├── config.lua           # 配置管理
├── utils.lua            # 工具函数
├── input_manager.lua    # 输入管理
├── scene_manager.lua    # 场景管理
├── resource_manager.lua # 资源管理
└── scene_base.lua       # 场景基类（可选，或者直接用表）

src/game/scenes/
├── test_scene_a.lua     # 测试场景 A（验证用）
└── test_scene_b.lua     # 测试场景 B（验证用）
```

### ✅ 验收标准
1. 启动游戏正常，控制台有 INFO 级别的启动日志
2. 按空格键可以在场景 A 和场景 B 之间切换
3. 切换场景时有日志输出，生命周期函数正常调用
4. 按 ESC 可以退出游戏
5. 键盘输入正常响应，按键有日志
6. 所有模块之间解耦清晰，没有循环依赖

### 💡 为什么这一版先做这些
- 这些是所有游戏都需要的基础能力，先搭好后面才能往上加东西
- 先验证核心链路通不通，再往上加复杂功能
- 这一版做完，框架的"地基"就稳了

## V0.2.0：渲染增强

### 🎯 目标
- 完善渲染相关的通用能力
- 实现分层渲染、相机系统、实体基类、动画系统
- 还是纯通用层，和具体游戏无关

### 📦 范围
- 渲染分层管理器
- 相机系统（位移、缩放、震动）
- 实体基类（所有游戏对象的父类）
- 动画与缓动系统（补间动画，多种缓动函数）
- 一个渲染测试场景

### 📁 涉及模块/文件
```
src/core/
├── render_manager.lua   # 渲染分层管理
├── camera.lua           # 相机系统
├── entity.lua           # 实体基类
└── animation.lua        # 动画与缓动系统

src/game/scenes/
└── render_test.lua      # 渲染测试场景
```

### ✅ 验收标准
1. 有 3-5 个渲染层，不同层的对象按顺序绘制
2. 相机可以移动、缩放，所有世界层的对象跟着动
3. 相机震动效果正常
4. 实体可以有位置、大小、颜色、缩放、旋转等属性
5. 可以创建补间动画，支持多种缓动函数
6. UI 层不受相机影响，固定在屏幕上
7. 测试场景里能看到明显的分层效果和动画效果

### 💡 为什么这一版先做这些
- 渲染是游戏的基础，先把渲染体系搭好，后面做棋盘棋子就容易了
- 这些都是通用能力，所有 2D 游戏都需要
- 先验证渲染管线没问题，再往上面加游戏内容

## V1.0.0：棋盘与棋子渲染

### 🎯 目标
- 做出能看的棋盘和棋子
- 可以点击选中棋子，有视觉反馈
- 从这一版开始进入象棋业务层

### 📦 范围
- 棋盘实体（木质感棋盘、格子、楚河汉界）
- 棋子实体（红黑双方，7 种棋子）
- 点击交互（选中/取消选中）
- 棋子移动（先不校验规则，能移动就行）
- 伪 3D 效果（阴影、轻微透视，营造纵深感）
- 主菜单场景（最简单的，能进游戏就行）

### 📁 涉及模块/文件
```
src/game/
├── entities/
│   ├── board.lua        # 棋盘实体
│   └── piece.lua        # 棋子实体
└── scenes/
    ├── main_menu.lua    # 主菜单场景
    └── game_play.lua    # 游戏对战场景
```

### ✅ 验收标准
1. 启动后进入主菜单，点"开始游戏"进入对战场景
2. 棋盘渲染正常，有 9 列 10 行，楚河汉界在中间
3. 32 个棋子初始位置正确摆放
4. 点击棋子有选中效果（高亮、抬升、阴影变化）
5. 选中一个棋子再点一个格子，棋子能移动过去（暂时不校验规则）
6. 视觉上有纵深感（阴影、分层），不是平面的
7. 按 R 可以重置棋盘
8. 按 ESC 返回主菜单

### 💡 为什么这一版先做这些
- 先把"看得见摸得着"的东西做出来，有成就感
- 先验证视觉效果，后面再加逻辑
- 有了可视化，后面调试规则也方便
- 符合"最小可运行"原则，这一版就能玩个"简化版象棋"（虽然没规则）

## V2.0.0：规则引擎

### 🎯 目标
- 实现完整的中国象棋规则
- 所有走法都符合规则，非法走法不能走
- 有将军、将死、和棋判定

### 📦 范围
- 规则引擎模块
- 7 种棋子的走法校验（将/帅、士/仕、象/相、马、车、炮、兵/卒）
- 特殊规则：马腿、象眼、炮架、九宫格限制、过河限制
- 将军检测
- 将死/困毙判定
- 飞将规则
- 和棋判定（简单版，比如 50 回合没吃子）
- 合法走法高亮显示

### 📁 涉及模块/文件
```
src/game/
└── rules.lua            # 规则引擎
```
（修改 game_play.lua，集成规则校验）

### ✅ 验收标准
1. 所有棋子走法都符合规则，非法走法会被拒绝
2. 选中棋子后，所有合法走法的格子会高亮
3. 将军时会有提示（文字提示就行）
4. 将死时游戏结束，显示胜负
5. 飞将规则生效（两个将不能在同一列直接对视）
6. 兵过河后可以左右走，没过河只能往前走
7. 象不能过河，象眼被塞住不能走
8. 马腿被别住不能走
9. 炮必须隔一个子才能吃子
10. 士和将不能出九宫格

### 💡 为什么这一版先做这些
- 规则是象棋的核心玩法，没有规则就不是象棋
- 核心玩法做好了，游戏才真正"能玩"
- 规则引擎是纯逻辑，和渲染分离，方便测试
- 这一版做完，就能正经下象棋了（虽然只有人人对战）

## V4.0.0：音效与动效

### 🎯 目标
- 提升游戏的手感和体验
- 加入音效、粒子特效、动画反馈
- 让游戏玩起来更"带感"

### 📦 范围
- 音频管理器（音效 + 背景音乐）
- 粒子系统管理器
- 棋子移动动画（不是瞬移，是平滑移动）
- 吃子动画和特效
- 选中特效
- 将军特效
- 游戏结束特效
- 各种音效（选中、移动、吃子、将军、结束等）
- 相机震动效果（增强打击感）

### 📁 涉及模块/文件
```
src/core/
├── audio_manager.lua    # 音频管理
└── particle_system.lua  # 粒子系统

assets/
├── sounds/              # 音效文件
└── music/               # 音乐文件
```
（修改 game_play.lua，集成动效和音效）

### ✅ 验收标准
1. 棋子移动有平滑的动画，不是瞬移
2. 吃子有爆炸粒子效果和震动
3. 选中棋子有光晕粒子效果
4. 将军有红色特效和震动
5. 游戏结束有庆祝/失败特效
6. 每个操作都有对应音效（选中、移动、吃子、将军、结束）
7. 有背景音乐，可以在设置里开关和调音量
8. 音效和音乐音量可以分别调节
9. 所有动效不影响游戏逻辑，只是视觉反馈

### 💡 为什么这一版先做这些
- 核心玩法做完之后，就是打磨体验
- 音效和动效是提升游戏"手感"最有效的方式
- 这些模块也是通用的，后面做其他游戏都能用
- 这一版做完，游戏的完成度和体验感会上一个大台阶

## V5.0.0：AI 对手与发布准备

### 🎯 目标
- 加入 AI 对手，支持人机对战
- 完善打包发布流程
- 达到可以发布的标准

### 📦 范围
- AI 模块（多种难度：简单、普通、困难）
- Minimax 算法 + 阿尔法剪枝
- 评估函数
- 主菜单加入人机对战选项和难度选择
- Steam 打包发布脚本
- Windows/macOS/Linux 桌面版打包
- Android APK 打包说明
- 性能优化和 Bug 修复
- 最终测试

### 📁 涉及模块/文件
```
src/game/
└── ai.lua               # AI 对手

scripts/
├── build_windows.sh     # Windows 打包脚本
├── build_macos.sh       # macOS 打包脚本
├── build_linux.sh       # Linux 打包脚本
├── build_android.sh     # Android 打包脚本
└── build_steam.sh       # Steam 打包脚本
```
（修改主菜单，加入人机对战和难度选择）

### ✅ 验收标准
1. 可以选择人机对战模式，选择难度
2. 简单难度：比较弱，随便下都能赢
3. 普通难度：有一定棋力，普通玩家需要认真下
4. 困难难度：比较强，普通玩家很难赢
5. AI 走棋有思考时间，不是瞬间走（更像真人）
6. Windows/macOS/Linux 都能打包成功，能运行
7. 有详细的打包发布文档
8. Steam 发布的流程和注意事项都写清楚
9. Android 打包的路径和坑点都写清楚
10. 游戏没有明显 Bug，流程完整

### 💡 为什么这一版最后做
- AI 是锦上添花的功能，不是核心玩法必须的
- 先把基础做好，再加 AI
- 发布相关的事情放在最后，等游戏稳定了再做
- 这一版做完，就是完整的可发布产品了

所有版本都会打 tag，随时可以切换验证：

