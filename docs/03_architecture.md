# 03. 整体架构

## 架构总览

采用**分层架构**，严格分离「通用 2D 游戏框架层」和「中国象棋业务层」。

核心设计原则：**通用层绝不依赖业务层，业务层只调用通用层提供的接口**。

这样设计的好处是：后续做其他 2D 游戏（比如弹珠、卡牌），只需要替换掉 game/ 目录的内容，整个 core/ 框架可以 100% 复用。

## 核心分层说明

### 1. 入口层（main.lua）
- 最薄的一层，只做"接线"工作
- 接收 LÖVE2D 的所有回调（load/update/draw/键盘/鼠标等）
- 转发给核心层的对应模块
- 类比：Web 开发中的 index.html 入口文件，或者 Spring Boot 的 main 方法

### 2. 通用框架核心层（src/core/）
这是整个项目最有价值的部分，完全与具体游戏无关，所有 2D 游戏都能复用。

#### 基础模块
| 模块 | 职责 | 类比 |
|------|------|------|
| logger.lua | 日志系统，分级输出，彩色日志 | Java 的 log4j / JS 的 console.log 封装 |
| config.lua | 配置管理，集中管理所有配置项 | Java 的 application.yml / Python 的 config.py |
| utils.lua | 工具函数，数学、字符串、表格等通用工具 | Lodash / Guava / Python 的 utils 模块 |

#### 系统模块
| 模块 | 职责 | 类比 |
|------|------|------|
| input_manager.lua | 输入管理，统一处理键盘/鼠标，支持按键映射 | JS 的事件监听封装 / Unity 的 Input 系统 |
| scene_manager.lua | 场景管理，场景切换、场景栈、生命周期管理 | React Router / Android 的 Activity 栈 |
| resource_manager.lua | 资源管理，图片/音效/字体的加载、缓存、释放 | Web 的资源预加载 / Unity 的 Resources 系统 |

#### 渲染模块
| 模块 | 职责 | 类比 |
|------|------|------|
| render_manager.lua | 分层渲染管理，按层排序，批量绘制 | CSS 的 z-index / Unity 的 Sorting Layer |
| camera.lua | 相机系统，位移、缩放、震动、跟随 | Unity 的 Camera / 游戏里的镜头 |
| entity.lua | 实体基类，所有游戏对象的父类，有位置、大小、渲染等 | Unity 的 GameObject / ECS 里的 Entity |
| animation.lua | 动画与缓动系统，补间动画，多种缓动函数 | GreenSock / CSS transition / 动画库 |
| particle_system.lua | 粒子系统，封装 LÖVE2D 原生粒子，特效预设 | Unity 的 ParticleSystem / 粒子特效库 |

#### 其他模块
| 模块 | 职责 | 类比 |
|------|------|------|
| audio_manager.lua | 音频管理，音效/背景音乐分离，音量控制，缓存 | Web Audio API 封装 / Unity 的 AudioSource |
| ui/ 目录 | 通用 UI 组件库，按钮、面板、复选框等 | React 组件库 / Element UI / 游戏 UI 框架 |
| save_manager.lua | 存档管理，保存/读取/删除游戏存档和设置 | localStorage / Unity PlayerPrefs / 存档系统 |

### 3. 游戏业务层（src/game/）
这一层是中国象棋特有的逻辑，换游戏就换掉这一层。

| 模块 | 职责 |
|------|------|
| entities/board.lua | 棋盘实体，渲染棋盘、处理格子坐标转换、管理棋子位置 |
| entities/piece.lua | 棋子实体，渲染棋子、保存棋子类型和颜色 |
| scenes/main_menu.lua | 主菜单场景 |
| scenes/game_play.lua | 游戏对战场景，核心玩法都在这里 |
| rules.lua | 象棋规则引擎，走法校验、吃子、将军、胜负判定 |
| ai.lua | AI 对手，不同难度的 AI 算法 |

### 4. 资源层（assets/）
所有静态资源：图片、音效、音乐、字体、配置文件等。

## 关键设计决策

### 决策 1：严格分层，通用层不依赖业务层
- **为什么**：保证框架的复用性，换游戏不用改核心层
- **怎么保证**：core/ 目录下的代码绝对不允许 require game/ 目录下的任何东西
- **验证方式**：把 game/ 目录删掉，core/ 依然能正常运行（只是没内容）

### 决策 2：场景驱动，每个场景是独立的世界
- **为什么**：不同场景（主菜单、游戏中、设置页）逻辑差异大，分开管理更清晰
- **怎么实现**：每个场景有完整的生命周期（load/enter/update/draw/exit/unload）
- **类比**：就像不同的网页，每个页面有自己的逻辑，通过路由切换

### 决策 3：实体 + 组件的简化版（不用 ECS）
- **为什么**：纯 ECS 对于小型 2D 游戏太复杂，单人开发没必要
- **怎么做**：用简单的继承式实体基类，需要什么功能就加什么方法
- **权衡**：牺牲一点灵活性，换来简单易懂、好维护

### 决策 4：分层渲染，不用深度测试
- **为什么**：2D 游戏用分层足够，简单直观，性能也好
- **怎么做**：预定义几个渲染层（背景/世界底层/世界中层/世界上层/UI层），每个实体指定所属层
- **类比**：就像 Photoshop 的图层，下面的层先画，上面的层后画，盖在上面

### 决策 5：资源统一管理，自动缓存
- **为什么**：避免重复加载同一张图片/音效，提升性能
- **怎么做**：ResourceManager 统一加载，加载过的资源存在缓存里，下次直接用
- **注意**：大资源（比如长音乐）可以选择不缓存，或者手动释放

## 数据流

### 启动流程
```
main.lua (love.load)
    → 初始化日志、配置
    → 初始化输入、资源、场景管理
    → 初始化渲染、相机、动画
    → 初始化 UI、音频、存档
    → 注册所有场景
    → 切换到初始场景（主菜单）
```

### 每帧更新流程
```
love.update(dt)
    → 输入管理器：更新输入状态
    → 场景管理器：更新当前场景
    → 动画系统：更新所有动画
    → 粒子系统：更新所有粒子
    → 音频系统：清理已播放完的音效
    → UI 系统：更新 UI 状态
```

### 渲染流程
```
love.draw()
    → 渲染管理器：按层顺序渲染
        → 背景层
        → 世界底层
        → 世界中层
        → 世界上层
        → UI 层（不受相机影响）
    → UI 系统：渲染 UI
```

## 扩展到其他游戏的方式

如果要做第二款游戏（比如弹珠台），只需要：
1. 新建 game_pinball/ 目录（或者替换 game/ 目录）
2. 实现弹珠台自己的场景、实体、规则
3. 完全复用 core/ 目录的所有通用能力
4. 不需要修改核心层的一行代码

这就是分层架构的价值——一次造轮子，多次复用。
