# 09. 主要风险与替代方案

## 为什么要做风险登记

独立开发不确定性很多，提前想到可能的风险和应对方案，到时候就不会慌。

**原则**：提前识别风险，提前准备替代方案，真遇到了就不会手忙脚乱。

---

## 风险列表

按风险等级从高到低排列：

---

### 风险 1：Lua 不熟悉，开发效率低

**等级**：高
**概率**：高
**影响**：开发慢，Bug 多，容易放弃

**描述**：
你说你不会 Lua，虽然熟悉其他语言，但 Lua 的语法、特性、API 都不熟，一开始可能会踩很多坑，开发效率低。

**应对方案**：
1. **注释里加类比**：所有 Lua 特性和 LÖVE2D API 都加注释，类比 JS/Java/Python，降低理解成本
2. **先做小版本**：每个版本都很小，做完就验证，不会走太远
3. **遇到问题查官方文档**：LÖVE2D 官方文档很全，示例也多
4. **代码尽量简单**：不用 Lua 的高级特性，就用最基础的语法，好理解好维护
5. **多用 print 和日志**：调试的时候打日志，看变量是什么值

**替代方案**：
- 如果实在觉得 Lua 难用，可以考虑换用其他语言的游戏框架，比如：
  - Phaser.js（JS，2D 游戏框架，Web 版）
  - Godot（GDScript，和 Python 很像，也可以用 C#）
  - Pygame（Python，简单易上手）
- 但是 LÖVE2D 的跨平台打包很方便，而且轻量，还是建议坚持一下，Lua 不难的，有其他语言基础的话一周就能上手

---

### 风险 2：架构设计不合理，后期不好扩展

**等级**：高
**概率**：中
**影响**：后面做其他游戏的时候要大改，复用性差

**描述**：
你希望这套框架能复用到其他 2D 游戏，但如果一开始架构设计不好，核心层和业务层耦合了，后面就很难复用，等于白做。

**应对方案**：
1. **严格分层**：core/ 和 game/ 严格分离，core 层绝不依赖 game 层
2. **每个版本做完都复盘**：看看架构有没有问题，及时调整
3. **不提前过度设计**：先做最简单能跑的，需要扩展的时候再重构，不要一开始就做"完美架构"
4. **做第二个游戏的时候验证**：等象棋做完了，做第二款游戏（比如弹珠）的时候，验证框架复用性，发现问题就改
5. **每个模块职责单一**：一个模块只做一件事，改一个地方不会影响其他地方

**替代方案**：
- 如果发现架构实在不合理，也可以考虑直接用现成的游戏框架（比如上面说的 Godot），不用自己造轮子
- 但是自己造轮子的好处是完全可控，想学东西的话还是值得的

---

### 风险 3："伪 3D"效果成本太高，做不出来想要的效果

**等级**：中
**概率**：中
**影响**：视觉效果达不到预期，打击信心

**描述**：
你想要"2D 但有纵深感"的效果，但是如果用纯 2D 做，可能效果有限，或者成本很高，做出来不好看。

**应对方案**：
1. **分层渲染**：背景层、棋盘层、棋子层、特效层，不同层有视差，就能有纵深感
2. **阴影**：每个棋子加阴影，抬高一点，就有立体感了
3. **透视错觉**：棋盘稍微有点透视变形（近大远小），不用真 3D，只是视觉上的错觉
4. **粒子特效**：爆炸、光晕这些粒子效果，能增加空间感
5. **相机运动**：镜头推拉、震动，也能增加空间感
6. **先做简单版**：先做基础的分层和阴影，效果不够再慢慢加，不用一步到位

**替代方案**：
- 如果 2D 效果实在不满意，可以考虑用 3D 引擎做 2.5D 游戏，比如 Godot 或者 Unity，把相机设成正交视角，其实也不晕
- 但是你说你会晕 3D，那还是先试试 2D 分层的方式，应该足够了，象棋不需要太复杂的效果

---

### 风险 4：规则引擎写不对，Bug 多

**等级**：中
**概率**：中
**影响**：游戏规则不对，没法玩

**描述**：
中国象棋规则说简单也简单，说复杂也复杂，很多特殊规则（马腿、象眼、飞将、困毙）容易写错，Bug 多。

**应对方案**：
1. **把规则模块独立出来**：rules.lua 单独一个文件，纯逻辑，不依赖渲染，方便测试
2. **写单元测试**：每个棋子的走法都写测试用例，覆盖各种情况
3. **逐步实现**：先实现简单的棋子（车、兵），再实现复杂的（马、炮、将）
4. **多玩多测**：写完就自己下几盘，试试各种情况
5. **找别人测**：写完了找会下象棋的人帮你测测

**替代方案**：
- 如果自己写规则引擎太麻烦，可以找现成的 Lua 象棋规则库，但是估计不多
- 或者参考开源的象棋游戏的规则代码，但是注意别直接抄，要理解了再写

---

### 风险 5：AI 太弱或者太强，不好玩

**等级**：中
**概率**：高
**影响**：人机对战体验差，没人玩

**描述**：
AI 难度不好把握，太简单了没意思，太难了又打不过，不好平衡。

**应对方案**：
1. **分多个难度等级**：简单、普通、困难，玩家自己选
2. **简单难度**：随机走，或者只看一步，肯定弱
3. **普通难度**：Minimax 2-3 层，有一定棋力
4. **困难难度**：Minimax 4-6 层 + 剪枝 + 开局库，比较强
5. **加入随机性**：同样的局面 AI 不一定走一样的，每次玩都不一样
6. **根据玩家水平调整**：后面可以加自适应难度，玩家赢多了就变强，输多了就变弱

**替代方案**：
- 如果自己写 AI 太麻烦，可以用现成的象棋 AI 引擎，但是 Lua 的估计不多
- 或者干脆不做 AI，只做人人对战，也能玩，AI 后面再加

---

### 风险 6：Steam 上架不顺利

**等级**：低
**概率**：中
**影响**：发布不了，或者审核很久

**描述**：
Steam 上架需要审核，可能会遇到各种问题，比如商店页不通过、游戏有问题等。

**应对方案**：
1. **提前看 Steam 上架指南**：先了解要求，提前准备好材料
2. **先上架 itch.io**：在 Steam 审核的同时，先在 itch.io 发布，有玩家玩
3. **商店页素材提前准备**：截图、宣传片、描述，都提前弄好
4. **游戏不要有违规内容**：象棋肯定没问题，不用担心
5. **耐心等审核**：审核一般几天到一周，不用急

**替代方案**：
- 如果 Steam 太麻烦，可以先只在 itch.io 发布，独立游戏很多都在上面
- 或者直接发压缩包，自己推广

---

### 风险 7：Android 打包太麻烦，做不出来

**等级**：低
**概率**：高
**影响**：移动端版本出不来

**描述**：
Android 打包比桌面版麻烦很多，需要 SDK、NDK、签名、适配各种机型，容易踩坑。

**应对方案**：
1. **优先级放低**：先做好桌面版，移动端后面再说
2. **先用简单的方式**：用现成的模板把游戏塞进去，先跑起来再说
3. **性能优化慢慢来**：先能运行，再优化性能
4. **参考官方文档和教程**：LÖVE2D 官方有 Android 移植的文档，跟着做

**替代方案**：
- 如果 Android 实在太麻烦，可以先不做移动端，只做桌面版
- 或者做 Web 版，手机浏览器也能玩，比原生 APK 简单

---

### 风险 8：项目太大，半途而废

**等级**：中
**概率**：中
**影响**：做不完，烂尾

**描述**：
独立开发很容易一开始热情很高，做着做着就累了，或者遇到难题卡壳，就放弃了。

**应对方案**：
1. **增量开发，每个版本都有成果**：每个版本做完都能玩，有成就感，更容易坚持
2. **先做最小可用版本**：先把核心玩法做出来，能玩就行，其他后面再加
3. **不要追求完美**：第一版粗糙一点没关系，慢慢迭代
4. **合理安排时间**：不要一下给自己定太大的目标，每天做一点
5. **经常玩自己做的游戏**：有正反馈，才有动力继续做

**替代方案**：
- 如果不想做完整的大项目，可以先做个小 demo，验证想法
- 或者先做功能少一点的版本，能发布就行，后面再更新

---

### 风险 9：性能问题，游戏卡顿

**等级**：低
**概率**：低
**影响**：游戏体验差

**描述**：
粒子太多、动画太多，可能会卡顿，尤其是低配置电脑或者手机上。

**应对方案**：
1. **控制粒子数量**：不要一下子搞几千个粒子，够用就行
2. **资源缓存**：图片音效都缓存起来，不要重复加载
3. **分层渲染优化**：不需要每帧都画的可以先画到 Canvas 上
4. **设置里加画质选项**：低配置可以关掉粒子特效
5. **性能检测**：看帧率，哪里卡优化哪里

**替代方案**：
- 如果性能实在不够，就减少特效，保证流畅最重要
- 象棋游戏对性能要求不高，一般不会有这个问题

---

### 风险 10：Lua 生态小，遇到问题不好搜

**等级**：低
**概率**：中
**影响**：遇到问题解决慢

**描述**：
Lua 不像 JS、Python 那么流行，遇到问题可能搜不到答案。

**应对方案**：
1. **官方文档是最好的老师**：LÖVE2D 的官方文档非常全，示例也多
2. **官方论坛和社区**：love2d.org 的论坛，还有 Discord 社区，很多人可以问
3. **GitHub 上搜开源项目**：很多人用 LÖVE2D 做游戏，参考别人的代码
4. **原理都是相通的**：游戏开发的原理都是一样的，只是语言不一样，会其他语言的话原理都能用

**替代方案**：
- 实在觉得 Lua 生态差，可以换更流行的语言和框架，比如 Godot 或者 Phaser

---

## 风险应对总原则

1. **提前识别，提前准备**：不要等问题发生了才想怎么办
2. **先做最简单能跑的版本**：不要一开始就追求完美，先有个东西能玩，再慢慢优化
3. **小步快跑，及时调整**：每个版本都复盘，发现问题及时改，不要攒到最后
4. **留好备用方案**：每个风险都有替代方案，A 方案走不通就走 B 方案，不会卡死
5. **接受不完美**：独立开发，不可能什么都做好，先完成再完美

---

## 最高优先级的三个风险

如果只能关注三个，那就是：
1. **Lua 不熟悉，开发效率低** → 多写注释，多做小版本验证
2. **架构不合理，复用性差** → 严格分层，做第二款游戏的时候验证
3. **项目太大，半途而废** → 增量开发，每个版本都有成就感，先完成再完美

这三个搞定了，其他的都是小问题。
