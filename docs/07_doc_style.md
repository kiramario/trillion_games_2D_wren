# 07. 文档规范

## 为什么要有文档规范

- 独立开发也需要文档，过几个月回来一看，全忘了
- 好的文档可以减少重复踩坑
- 如果以后找人协作或者开源，文档是门面
- 代码是写给机器看的，文档是写给人看的（包括未来的自己）

---

## 文档分类

### 1. 规划类文档（docs/ 目录下，0开头）
- 项目开始前写的，定方向、定架构、定计划
- 一般不会频繁变动
- 包括：愿景、开发方式、架构、目录设计、路线图、规范等

### 2. 变更类文档
- changelog.md：每个版本的变更记录
- release_notes/：每个版本的发布说明（可选）

### 3. API 文档（可选，后面再加）
- 每个模块的 API 说明
- 自动生成或者手动写
- 项目小的时候可以先不写，等变大了再加

### 4. 开发文档
- 开发流程、调试技巧、常见问题
- 踩坑记录

---

## 文档命名

### 规划类文档
- 数字编号 + 下划线 + 英文简称
- 按编号排序，方便按顺序阅读
- 编号从 01 开始

示例：
```
01_vision.md
02_methodology.md
03_architecture.md
04_directory.md
05_roadmap.md
```

### 其他文档
- 全小写，下划线分隔
- 名字能看出内容是什么

---

## 文档格式

### 用 Markdown
- 所有文档都用 Markdown 格式
- 兼容性好，GitHub、VS Code 都能直接看
- 语法简单，写起来快

### 标题层级
- 每个文档只有一个一级标题（#），是文档的标题
- 二级标题（##）是大的章节
- 三级标题（###）是小节
- 最多到四级（####），不要太深

✅ 正确：
```markdown
# 文档标题

## 第一章
### 第一节
#### 第一点

## 第二章
```

### 列表
- 无序列表用 `-`
- 有序列表用 `1.`
- 列表项如果比较短，就一行
- 列表项如果有解释，可以换行缩进

### 表格
- 内容对比用表格，清晰直观
- 表头对齐，看起来整齐

### 代码块
- 代码用 ``` 包裹，指定语言
- Lua 代码就写 ```lua
- Shell 命令就写 ```bash

✅ 正确：
```lua
local function hello()
  print("hello world")
end
```

```bash
love .
```

### 强调
- 重要的内容用 **加粗**
- 特别重要的可以用 > 引用块或者加 emoji 突出
- 不要滥用加粗，太多了等于没有

## 内容规范

### 先总后分
- 每个大章节开头先总述，讲这章讲什么
- 然后再分点详细说
- 让人看开头就知道这章有没有自己想要的内容

### 通俗易懂
- 尽量用大白话，不要故作高深
- 技术术语第一次出现可以解释一下
- 可以用类比，用大家熟悉的东西做比方
- 目标是：让刚接触 Lua 的人也能看懂

### 面向初学者
- 因为你说你不会 Lua，所以文档尽量对初学者友好
- 涉及 Lua 特性的地方，可以加一句类比（"类似 JS 里的 XXX"）
- 涉及 LÖVE2D API 的地方，可以简单说明一下是做什么的

### 实用导向
- 文档是拿来用的，不是拿来凑字数的
- 不说空话套话
- 每一段都要有信息量
- 能举例子的地方就举例子

### 及时更新
- 代码改了，对应的文档也要同步更新
- 过时的文档比没有文档还坑人
- 每个版本更新的时候，同步更新 changelog 和相关文档

## 各个文档的内容要求

### README.md
项目的门面，第一个看的文档，必须包含：
1. 项目是什么
2. 怎么安装运行
3. 项目结构大概是什么样
4. 怎么切换版本
5. 文档索引（各个文档的链接）
6. 设计理念或者特点

### 01_vision.md（项目愿景）
- 项目定位
- 短期/中期/长期目标
- 设计理念
- 目标用户
- 成功标准

### 02_methodology.md（开发方式）
- 推荐什么开发方式
- 为什么选这个，为什么不选其他的
- 怎么落地到本项目

### 03_architecture.md（架构）
- 整体架构图（文字版或者 ASCII 图）
- 分几层，每层的职责是什么
- 核心模块说明
- 关键设计决策和理由
- 数据流
- 怎么扩展到其他游戏

### 04_directory.md（目录设计）
- 整体目录结构
- 每个目录是做什么的
- 每个主要文件是做什么的
- 目录设计原则
- 后续扩展建议

### 05_roadmap.md（版本路线图）
- 版本总览表
- 每个版本的详细说明：
  - 目标
  - 范围
  - 涉及的模块/文件
  - 验收标准
  - 为什么这一版做这些

### 06_code_style.md（代码规范）
- 命名规范
- 代码格式
- Lua 特定规范
- 模块规范
- 注释规范
- 最佳实践
- 提交规范

### 07_doc_style.md（文档规范）
- 为什么要有文档规范
- 文档分类
- 命名规范
- 格式规范
- 内容规范
- 各个文档的要求（就是这一节）

### 08_release.md（发布路径）
- 各个平台的发布方式
- 打包流程
- 工具和脚本
- 注意事项和坑点
- Steam 发布流程
- Android 打包流程

### 09_risks.md（风险登记）
- 有哪些风险
- 风险等级
- 发生概率
- 影响程度
- 应对方案
- 替代方案

### changelog.md（变更日志）
- 每个版本的变更记录
- 按版本倒序排列（最新的在最上面）
- 每个版本包含：
  - 版本号和日期
  - 新增了什么
  - 修改了什么
  - 修复了什么
  - 已知问题（可选）

## 写文档的小技巧

### 1. 先写大纲再填内容
- 先把一二级标题列出来
- 知道要写什么，再往里填内容
- 不容易写乱

### 2. 想到就写，再慢慢完善
- 不用追求第一次就写得完美
- 先把要点记下来，后面再整理润色
- 边做边更新，比最后一次性补容易多了

### 3. 多写例子
- 纯文字描述干巴巴，不如一个例子清楚
- 能举例子的地方就举
- 代码、截图、表格，都是好例子

### 4. 写给三个月后的自己看
- 写的时候想象一下，三个月后的你回来看这个文档，能不能看懂
- 不要默认读者知道你现在知道的所有事情
- 背景信息、为什么这么做，都可以写

### 5. 保持文档和代码同步
- 改了代码，顺手把文档也改了
- 不要攒着，攒多了就不想改了
- 提交代码的时候，文档也一起提交

---

## 文档工具

- 编辑器：VS Code 就很好用，自带 Markdown 预览
- 画图：可以用 ASCII 字符画，或者用 draw.io / Excalidraw 画图导出图片（后面需要的话再加）
- 目前先都用文字，简单直接，不用依赖额外工具

---

## 总结

文档的价值怎么强调都不过分。

**记住：代码是写给机器的，文档是写给人的（包括未来的自己）。**

现在多花半小时写文档，未来可能省半天时间找 bug、回忆当初为什么这么写。
