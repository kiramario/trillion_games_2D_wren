## 完成的功能

### 规则模块新增
1. **将帅对脸（飞将）规则**
   - 两个将不能在同一条竖线上直接对面
   - 任何导致将帅对脸的走法都是非法的

2. **自将判断**
   - 走子后不能让自己的将被将军
   - 走子前模拟走一遍，检查会不会被将军
   - 包括将军和将帅对脸两种情况都算

3. **将死判定**
   - 被将军，且没有任何一步棋可以解将
   - 判定为输，对方获胜

4. **困毙判定**
   - 没被将军，但所有棋子都没有合法走法
   - 中国象棋困毙算输，对方获胜

### 游戏状态更新
- `status` 字段：playing / checkmate / stalemate

- `undo_move()` 悔棋功能：
  - 回退一步
  - 恢复被吃掉的棋子
  - 恢复回合
  - 重置游戏状态

## 技术要点

### 1. 模拟走子判断自将
判断走子后会不会被将军，用的是模拟走子法：
1. 临时把棋子移到目标位置
2. 把目标位置的棋子标记为死亡
3. 检查是否被将军
4. 恢复原状

因为棋盘很小，这种方法性能完全没问题，而且逻辑简单不容易错。

类比：数据库事务，先改，检查没问题就提交，有问题就回滚。

### 2. 将死/困毙的判定逻辑
- 将死：被将军 + 没有任何合法走法
- 困毙：没被将军 + 没有任何合法走法

遍历己方所有棋子，每个棋子调用 `get_valid_moves()`，只要有一个棋子有至少一步合法走法，就没被将死/困毙。

### 3. 悔棋的实现
每走一步都把完整信息存在 `move_history` 里：
- 走的是哪个棋子
- 从哪走到哪
- 有没有吃子，吃的是哪个
- 当时是哪个回合

悔棋的时候全部还原就行，很简单。

### 4. 规则模块和游戏状态分离
规则模块是纯函数式的，输入棋子和游戏状态，输出是否合法。
游戏状态负责管理数据，调用规则来判断。
这样解耦，后面做AI的时候可以直接复用规则模块，不用改游戏状态的代码。

## 踩过的坑

### 坑1：模拟走子后忘记恢复
一开始模拟走子检查自将，检查完之后忘记把棋子位置恢复了，结果走子就乱了。
**解决：** 仔细写恢复逻辑，每一个修改都对应一个恢复操作，顺序也不能错。

### 坑2：悔棋时被吃的棋子没加回pieces列表
一开始悔棋只恢复了board数组里的棋子，没加回pieces列表，导致棋子总数不对，绘制也有问题。
**解决：** 悔棋时被吃的棋子要 `table.insert(self.pieces, captured_piece)` 加回去。

### 坑3：将死判定的赢家搞反了
一开始以为被将军的那一方赢，结果搞反了。
**解决：** 下一个回合的人被将死，所以当前走棋的人赢。比如红方走了一步，然后检查黑方是不是被将死，是的话红方赢。

### 坑4：game_over的时候还能走棋
游戏结束后还能继续点棋子走，逻辑不对。
**解决：** 鼠标点击处理最前面加个判断，game_over就直接return。
